## 0. 目的・位置づけ

**産業用ルーター／IoTゲートウェイの設置業者（SIer・施工者）向けの設置ナビ。**
必須5項目＋条件付き1項目（MIMO時のみ）＋任意1項目（現場名・印刷用）を入力するだけで、(1)推奨アンテナ構成、(2)設置起因のdB損失見積り（実測参考値）、(3)現場で守るべき警告・チェックリスト、(4)A4一枚の設置指示書（印刷）を出力する。

狙いは2つ：
1. **「つながらない」クレームの設置起因分を出荷前・施工時に潰す**（金属直貼り・盤内設置・防水不良・曲げ半径violation・MIMO片系統未接続 が五大要因＝v2.0の四大要因にMIMO延長本数不足を追加）。
2. **カタログ値ではなく実測データで語る**。数値の出典クラス（実測／換算／一般目安）を画面上で明示し、営業デモの説得力と技術的誠実さを両立する。

### スコープ外（明記事項）
- リンクバジェットの合否判定はしない（基地局距離・回線条件が不明のため）。出すのは「設置起因のマイナス要因の合計」まで。
- 電測値の自動取得・保存機能なし（印刷のみ）。多言語対応なし（日本語のみ）。
- 個別企業名・ブランド名・顧客名は一切表示しない（§10 表記ルール参照）。

## 3. 入力仕様

| # | 項目 | 型 | 選択肢（value） | 既定値 |
|---|---|---|---|---|
| 1 | 通信方式 | radio | LPWA・LTE-M・Cat.1 bis（`single`）／LTE Cat.1・Cat.4（`mimo`）／5G Sub6（`5g`） | `single` |
| 2 | 設置環境 | radio | 屋内（`indoor`）／屋外・雨がかり（`outdoor`）／金属盤・キュービクル内に通信機器（`cabinet`） | `indoor` |
| 3 | アンテナ設置面 | radio | 非金属（`nonmetal`）／金属＋スペーサー2〜8cm（`metal_spacer`）／金属直貼り（`metal_direct`） | `nonmetal` |
| 4 | 装置〜アンテナ良好地点の距離 | radio | 〜5m・標準ケーブル最長で届く（`std`）／+3m延長・〜8m（`ext3`）／+10m延長・〜15m（`ext10`）／+13m連結延長・〜18m（`ext13`） | `std` |
| 5 | 主要バンド | select | 800MHz帯（`b800`）／900MHz帯・920MHz LPWA含む（`b900`）／1.5GHz帯（`b1500`）／1.7GHz帯（`b1700`）／2GHz帯（`b2000`）／3.5GHz帯以上・5G（`b3500`）／**不明・安全側で計算（`unknown`）** | 方式連動（下記）。`unknown`は既定にしない |
| 6 | MIMO 2本間隔20cm確保 | radio | はい・2本構成も検討（`true`）／いいえ・一体型一択（`false`） | `true`。**`mimo`選択時のみ表示** |
| 7 | 現場名 | text | 自由入力 | 空 |

### 項目ラベル・注記（UI表示文言）
- 距離：フォーム脇に「現地未訪問で距離が不明な場合は、長め（+10m延長）を仮選択して持参し、現地で最短構成に切り替えてください」を表示。
- バンド：フォーム脇に「ルーター管理画面のBand表示で確認できます。不明な場合は『不明』を選択すると安全側（悪いほう）の見積りになります」を表示。ラベルは「契約回線の周波数帯」。
- MIMO間隔：質問文を「シングルアンテナ2本を20cm離して設置できるスペースがありますか？（一体型MIMOを使う場合はこの制約はありません）」とする。§4Aで一体型MIMOは常に第一推奨、`true`のときのみ「シングル2本構成も可」を備考に追加する（既定値`true`は「まず両方の選択肢を見せる」ため）。
- `cabinet`（盤内）選択時は「設置面」欄の脇に「アンテナを盤外へ出した後の設置面を選んでください（盤内に置く前提の入力はしません）」を表示。

### 入力の連動ルール（UI側）
- 方式変更時にバンド既定値を自動セット：`single`→`b900`、`mimo`→`b800`、`5g`→`b3500`。**ただしユーザーがバンドを手動変更した後は、通常は上書きしない。例外**：現在の方式が`5g`から他方式へ変わり、かつ選択中のバンドが`b3500`の場合に限り、`b3500`は選択不可（disabled）になるため方式既定バンドへ強制的にリセットし、UI上に一時通知（例：「バンドを900MHz帯にリセットしました」）を出す。同様に、方式が`mimo`から他方式へ変わったときは`mimoSpacingOk`を`null`に戻す。
- `b3500` は `5g` 選択時のみ選択可能（他方式では disabled＋「対応範囲外」注記）。小型シングル/小型MIMOの対応周波数は814〜960/1428〜2170MHzのため。
- `unknown` はどの方式でも選択可能（disabledにしない）。
- **プリセット適用時**：手動変更フラグを一旦リセットしてからプリセットのinputをフォームへ反映する（以後は通常の連動ルールに従う）。

## 4. 出力仕様

### A. 推奨構成カード
| 項目 | 内容 |
|---|---|
| 製品カテゴリ | 小型シングル／小型MIMO一体型／5G Sub6対応MIMO（表D参照） |
| ケーブル長 | `single`/`mimo`: 2.5m／5m、`5g`: 1m／2.5m／5m。「良好地点に届く最短を選ぶ（余長最小）」を添える |
| 取付方法 | 設置面から自動選択（金属面→ネオジム磁石を第一候補・M4ねじ併記、非金属面→両面テープ第一候補・M4ねじ併記） |
| 延長ケーブル | `ext3`→低損失防水延長3m、`ext10`→同10m、`ext13`→3m+10m連結。**数量**：`mimo`/`5g`選択時は「×2本（系統ごとに1本、同一長で揃える）」、`single`は「×1本」 |
| 備考 | `mimo`＋間隔OK→「シングル2本を20cm離隔する構成も可」、`5g`→「LTEでも使用可・将来の5G移行でアンテナ交換不要」 |

印刷時の推奨構成要約ブロックは以下5項目に固定（他フィールドは印字しない）：製品カテゴリ／対応周波数／ケーブル長（選択優先1つ）／取付方法（第一候補のみ）／延長（label・型番・数量。`std`時は非表示）。

### B. dBバジェット表（本アプリの中核）
行構成（該当行のみ表示、掲載順は下表の並び順で固定）：

| 行 | 表示条件 | 値 | 出典クラス |
|---|---|---|---|
| 金属直貼りペナルティ | `metal_direct` | 表Aのバンド値。`band==unknown`は方式の対応範囲内で最も損失が大きいバンド（`single`/`mimo`は800MHz帯、`5g`も800MHz帯＝金属損失は低周波ほど大きいため）を安全側採用し○クラスへ降格。`band==b3500`は範囲外表示（下記） | ●実測参考値（`unknown`時は○） |
| 金属面＋スペーサー | `metal_spacer` | **レンジ表示「0〜（同条件の直貼り値）の間・軽減効果の定量実測なし」**＋仮設置確認必須 | ○一般的な目安 |
| 延長ケーブル損失 | `distance != std` | 表Bの当該バンド換算値（`band==b900`のみ●実測、他は◐換算）。`band==unknown`は方式の対応範囲内で最も損失が大きいバンド（`single`/`mimo`は2GHz帯、`5g`は3.5GHz帯＝ケーブル損失は高周波ほど大きいため）を採用し○クラスへ降格 | ◐（`unknown`時は○） |
| 盤内 | `cabinet` | ハイライト枠（rowsには含めない。表の直下に独立ブロックとして描画。印刷でも同位置）。文言は動的：延長を選択済みなら「延長損 約{選択中の延長行の値}dB を払って外に出す方が、盤内の遮蔽損（一般に十数dB以上・環境依存の一般値）を避けられ大幅に有利」。`distance==std`（延長未選択）なら仮定形「例えば+3m延長でも損失は約−2dB@900MHz程度に収まり、盤内の遮蔽損（一般に十数dB以上）を避ける方が有利なケースが多い」 | ○一般的な目安 |
| 合計 | 常時 | 下記の優先順位ルールで決定 | 最も弱いクラスを継承（合計テキストに記号を埋め込み済みで返す。追加フィールドは持たない） |

**合計行の決定規則（優先順位付き）**：
1. スペーサー行（レンジ）が存在する → レンジ和「約−X〜−YdB」を表示。
2. レンジ行はないが、範囲外（`valueDb==null`）の行が存在する（`b3500`＋`metal_direct`のケース） → 数値行の和に「＋実測範囲外の項目あり（数値化不可、必ず仮設置で確認）」を付記。ゼロ扱いにしない。
3. 数値行・レンジ行・範囲外行のいずれも存在しない → 「設置起因の損失なし（0dB）」。

- `b3500`＋`metal_direct`（かつ`band`が`unknown`でない）は「実測データ範囲外（〜2.17GHzまで実測）。高周波では傾向が異なるため必ず仮設置で確認」を表示し、数値は出さない（`valueDb: null`）。
- 金属ペナルティの実測は小型シングル1機種によるもの。`mimo`/`5g`選択時は行末に「※実測は小型シングルによる傾向値」を付す（一体型MIMOは金属上の影響を抑えた設計だが定量値は未取得＝○扱い）。
- `band==unknown`のとき常時警告W14（下記）を表示。
- 合計行の直下に固定文言：「圏内/圏外の判定は本ナビでは行いません。仮設置での電測確認が最終判断です。」

### C. 警告リスト（該当時のみ）
表示順は下表の並び順（must→recommend、同severity内はID昇順）で固定。エンジンが順序を保証し、UIは並べ替えない。

| ID | 発火条件 | 種別 | 文言要旨（正確な文言はエンジン正本） |
|---|---|---|---|
| W1 | `outdoor` または `cabinet` | 必須 | コネクタ部はIPX6適用外。自己融着テープ＋絶縁テープで防水処理（`cabinet`起因の場合は「延長先が屋外・雨がかりの場合」を付記） |
| W2 | `outdoor` または `cabinet` | 必須 | 排水穴に水が入らない姿勢で設置（ケーブル下向き推奨）（`cabinet`起因の場合は同上の付記） |
| W3 | `metal_direct` | 推奨 | 周囲金属から四方5cm以上離隔、金属面の高さがアンテナを超えないこと。スペーサー2〜8cmで改善余地 |
| W4 | `metal_spacer` | 推奨 | スペーサー厚2〜8cm目安（環境依存）。仮設置で通信品質確認 |
| W5 | 常時 | 必須 | 曲げ半径2.1cm以上（可動部3.0cm）・コネクタ直後2cmは直進・SMA先端から70mm以上確保して90°曲げ・余長は直径12cm以上で巻く |
| W6 | 常時 | 必須 | SMA締付トルク6〜10kgf・cm（トルクレンチ使用）。ピン先端に触れない（静電気・変形） |
| W7 | `mimo`＋間隔NG | 推奨 | 20cm確保不可→一体型MIMO推奨。代替：高さをずらす／平行にならないよう角度を変える |
| W8 | `cabinet` | 必須 | 盤内アンテナは実質NG。延長ケーブルで盤外の良好地点（窓際・屋外）へ出すこと |
| W9 | `cabinet`＋`std` | 必須 | 標準ケーブルのみで盤外の良好地点に届くか要確認。届かない場合は+3m/+10m延長、それでも不足なら+13m連結を選択（W10参照） |
| W10 | `ext13` | 推奨 | 連結は可能だが損失増（@900MHz実測で合計約−5dB。選択バンドでの換算値はdBバジェット表参照） |
| W11 | `surface==nonmetal`（両面テープが取付第一候補となるため） | 推奨 | 15℃以下では被着面とテープを温めて貼付。再貼り付け不可 |
| W12 | `b3500`かつ`5g`以外 | 必須 | 選択バンドは製品対応範囲外（UI側でdisabledだがエンジンでも防御） |
| W13 | `(mimo または 5g)`かつ`distance != std` | 推奨 | 延長ケーブルは系統ごとに1本＝合計2本必要。2本は同一長で揃えること |
| W14 | `band==unknown` | 必須 | バンド不明選択時の表示値は安全側（最悪ケース）の見積りです。実際のバンドが判明次第、選び直して再計算してください |

### D. 設置指示書（印刷・A4縦1枚）
1. **ヘッダ**：タイトル＋記入欄（罫線）：現場名・日付・施工者・装置番号／シリアル・回線番号（SIM）
2. 推奨構成（§4A末尾で定義した5項目の要約ブロック）
3. dBバジェット表（コンパクト版・出典クラス記号付き）
4. **準備物**（条件付きブロック。該当時のみ列挙）：
   - 常時：トルクレンチ（SMA用）
   - `outdoor`または`cabinet`：自己融着テープ＋絶縁テープ
   - `metal_spacer`：樹脂スペーサー（2〜8cm）
   - `distance != std`：延長ケーブル（型番×数量。§4A参照）
5. **設置チェックリスト**（チェックボックス付き。基本7項目＋条件付き最大4項目）：

   基本（常時表示）：
   - □ アンテナは縦向き（長手方向を地面と垂直・ケーブル下出し）
   - □ できるだけ高所・見通しの良い場所
   - □ ケーブル：コネクタ直後2cm直進・曲げ半径2.1cm以上・余長は直径12cm巻き
   - □ SMAトルク6〜10kgf・cm（トルクレンチ使用）・ピンに触れない
   - □ 仮設置で通信品質を確認してから本固定
   - □ 電測値を下欄に記録（設置前後）
   - □ 記入後は本紙を撮影し施工報告に添付

   条件付き（該当時のみ追加）：
   - □ アンテナを盤外の良好地点に延長ケーブルで設置した（`cabinet`）
   - □ 周囲金属から四方5cm以上（金属面はスペーサー2〜8cm検討）（`metal_direct`または`metal_spacer`）
   - □ コネクタ防水処理（自己融着＋絶縁テープ）・排水穴に水が入らない姿勢（`outdoor`または`cabinet`）
   - □ 間隔20cm確保（不可なら一体型MIMOを使用）（`mimo`）

6. **電測記録欄**：設置前（仮設置時・良好地点）／設置後 の2列×測定日時／測定手段（ルーター管理画面／測定器）／RSRP・RSRQ・SINR・RSSI(参考)の記入欄。
   **目安（○一般的な目安・環境依存）**：RSRP −90dBm以上＝良好／−90〜−105dBm＝可・要再検討／−105dBm未満＝設置位置の再検討を推奨。SINR 13dB以上＝良好。RSSI（LPWA機種）は取扱説明書のしきい値を参照。
   ※「つながらない」クレーム時のエビデンスとして機能させる＝本アプリの核心価値。
7. フッタ：免責文（§10）

## 5. データ正本（エンジンに埋め込む定数。数値の発明禁止）

### 出典クラス
- ●実測参考値：出典資料の実測表・実測記載から転記した値
- ◐換算目安：●を起点に明示した式で換算した値（式もUIに表示）

## 6. 判定ロジック（決定表・計算なし、テーブル参照とif文のみ）

```
製品 = { single: 小型シングル, mimo: 小型MIMO一体型, 5g: 5G Sub6対応MIMO }[method]
延長本数 = (method == mimo || method == 5g) ? 2 : 1

延長 = { std: なし, ext3: EXT-3M, ext10: EXT-10M, ext13: EXT-3M+EXT-10M連結 }[distance]

取付 = surface が metal_* ? [磁石, M4ねじ] : [両面テープ, M4ねじ]

実効バンド(用途) = band != unknown ? band
                 : 用途が金属行 ? (method==5g ? b800 : b800)   // 低周波ほど金属損失が大きい＝両ケースともb800
                 : 用途が延長行 ? (method==5g ? b3500 : b2000) // 高周波ほどケーブル損失が大きい＝方式内最大バンド
実効クラス(unknown時) = "general"（○。band!=unknownなら通常クラス）

金属行 = surface == metal_direct ?
           ( 実効バンド(金属行) == b3500 ? {valueDb: null, text: "実測範囲外"} : {value: 表A[実効バンド(金属行)], cls: 実効クラス(unknown時)} )
       : surface == metal_spacer ?
           ( 実効バンド(金属行) == b3500 ? {rangeDb: null, text: "実測範囲外（スペーサーの効果も含め評価不可）"} : {rangeDb: [0, 表A[実効バンド(金属行)]]} )
       : なし

延長行 = distance != std ? { valueDb: 表B[distance][実効バンド(延長行)], cls: (band==b900 ? "measured" : 実効クラス(unknown時) || "converted") } : なし

盤内ハイライト = environment == cabinet ?
    { text: distance!=std ? "延長損 約{延長行.valueDb}dB を払って外に出す方が有利" : "例えば+3m延長でも損失は約−2dB@900MHz程度に収まり、盤内の遮蔽損を避ける方が有利なケースが多い" }
  : null

警告(cabinet/outdoor) = environment == cabinet ? (W1,W2[延長先屋外注記],W8, distance==std?W9)
                       : environment == outdoor ? (W1,W2) : なし

MIMO = method == mimo ? (mimoSpacingOk ? 備考「2本構成も可」 : W7) : (mimoSpacingOkは常にnullに正規化)
MIMO本数警告 = (method==mimo || method==5g) && distance!=std ? W13 : なし
バンド不明警告 = band==unknown ? W14 : なし

合計 = §4B「合計行の決定規則」参照（優先順位: レンジ和 > 範囲外あり注記付き和 > 数値行ゼロなら0dB）
```

- エンジンは**全域関数**：どの入力組合せでも例外なくResultを返す。
- **組合せ数**：`mimoSpacingOk`は`method==mimo`のときのみ`true/false`の2値、それ以外は`null`固定。よって総数＝`(method:mimo→1構成×2)+(method:single/5g→2構成×1)`のパターンを他4軸（environment3×surface3×distance4×band7）に掛けたもの＝ `(1×2 + 2×1) × 3×3×4×7 = 4 × 252 = 1008`通り（v2.0はband6択だったため864、v2.1で`unknown`を追加し7択になったため1008に変更。§8のSELFTEST件数もこれに合わせる）。
- 矛盾防止の不変条件：`metal_direct`の合計 ≦ `metal_spacer`の下限 ≦ `nonmetal`（符号付きdB値で比較。損失が大きいほど値は小さい）／`ext13` < `ext10` < `ext3` < `std`（同）。**`band==b3500`かつ金属直貼り／スペーサーの組合せは数値がnullのため、この不変条件の検証対象から除外**し、代わりに「valueDb/rangeDbがnullのとき表示文言に『範囲外』を含む」をSELFTESTで検証する。
- `mimoSpacingOk`に`method==mimo`以外で`true/false`が渡された場合は`null`として扱う（エンジン側で正規化）。`method==mimo`で`null`が渡された場合は保守的に`false`（間隔NG＝W7発火）として扱う。enum外の値が渡された場合は例外を投げ、§1の劣化動作に委ねる。

## 8. 自己テスト仕様（SECTION 5・T-6）

`file://` で開いた際に自動実行し、結果を`console`に出力する。**画面バッジは既定で非表示**（display:none）とし、以下のいずれかの場合のみ表示する：(a) URLの`location.hash`に`selftest`を含む、(b) 検証に1件でも失敗がある場合（この場合は常時・強制表示）。バッジDOM要素`<div id="selftest-badge">`はSECTION 5（T-6）が生成しbody末尾に追加する。SECTION 1（U-5）は`#selftest-badge`に対する印刷非表示CSS（`@media print { #selftest-badge { display: none !important; } }`）のみを書き、マークアップ自体（SECTION 2）には要素を置かない。

**スタブ段階の挙動**：`NAVI_ENGINE.VERSION`が`"stub"`で始まる間（T-6a〜U-5開発中）、SELFTESTは「STUB MODE」バッジ（強制表示）を出し、全域性・不変条件・スナップショット検証をスキップする。T-6bが本実装を完了しVERSIONを`"stub"`以外（例：`"1.0.0"`）に更新した時点で通常のSELFTESTが有効化される。

1. **全域性**：1008通り全input（`method!=mimo`のときは`mimoSpacingOk:null`固定の1通りのみ、`method==mimo`のときは`true`/`false`の2通り）で`compute`が例外なく、必須フィールドがすべて非空。
2. **不変条件**：§6の損失大小関係（`band==b3500`は金属系不変条件の対象外、範囲外文言の存在チェックで代替）／`b3500`×非5gでW12／`cabinet`でW8＋highlight非null／`outdoor`と`cabinet`でW1・W2／`(mimo||5g)`＋延長ありでW13／`unknown`でW14。
3. **プリセットスナップショット**：3プリセットの期待値（合計dB文字列・製品カテゴリ・警告ID集合）を固定値で照合。
4. **スタブフィクスチャ互換性**：T-6a作成のスタブ最低5フィクスチャ（§11参照）と、本実装の対応する入力での出力が同じ形状（フィールド欠落なし）であることを確認。

### T-6aスタブエンジンの仕様（最低5フィクスチャ）
`compute(input)`は入力に応じて以下の固定Resultを返す（本実装と同じフィールド形状で、値は仮のものでよい）：
1. `environment=="cabinet"` → P1相当（`highlight`非null、`extension`非null・`quantity:1`、`warnings`にW1/W2/W8/W9相当、`budget.rows`に金属行＋延長行）
2. `surface=="metal_direct" && environment!="cabinet"` → P2相当（`extension`非null・`quantity:2`、`warnings`にW7/W13相当、`highlight`はnull）
3. `surface=="metal_spacer"` → `rangeDb`非null（レンジ表示）のフィクスチャ
4. `band=="b3500" && method=="5g" && surface=="metal_direct"` → `valueDb:null`（範囲外表示）のフィクスチャ
5. 上記以外（既定） → P3相当（`warnings`空配列、`budget.rows`空、`highlight`null、`extension`null、合計「損失なし（0dB）」）

これによりU-5はnullable全分岐（highlight有無・extension有無と数量・rangeDb・valueDb null・警告0件と多数）を統合前に開発・確認できる。

### 統合手順（直列統合・コンフリクト回避）
本プロジェクトの視覚回帰ブランチと同様、**並列編集ではなく直列統合**を採用する（並列は同一ファイルの隣接編集でコンフリクトを生みやすいため）。
1. T-6a：スケルトン＋スタブをfeature由来の作業ブランチへコミット（この時点でセクション所有権・バッファ位置が確定）。
2. U-5：T-6aのブランチから分岐し、SECTION 1/2/4のみを実装（3/5・マーカー行・バッファ行には触れない。ファイル全体の再整形やフォーマッタ適用も禁止）。完成後、統合用ブランチへマージ。
3. T-6b：U-5マージ後の統合用ブランチから分岐し、SECTION 3/5のみを実装してスタブを置き換える（1/2/4には触れない）。完成後、統合用ブランチへマージ。
4. T-6c：統合後、Chrome実機で3プリセット・印刷・SELFTEST（VERSIONが`stub`でなくなっていること）を確認してからpush（push前にユーザー確認、既定ルール通り）。
